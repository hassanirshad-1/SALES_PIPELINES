1. What This System Does
The standard cold outreach flow is: send email → wait for reply → show demo to the small % who respond. This system inverts that. Every lead gets a personalized demo in the first email — before they reply to anything. The demo is a 2–3 page mini-website that looks custom-built for their business, swapping in their logo, company name, and niche-specific copy from a template.
This works because:
Seeing a working demo of your own business immediately proves capability
It stands out in an inbox full of text-only cold emails
Even a prospect who won't reply will click a link to see their own site
The CTA on the demo page itself ("Claim this site / Let's talk") does a second round of conversion
Core principle
Front-load the demo. Every lead sees a tangible, personalized preview of what you can build
for them before they even reply. You prove you can build for them before asking for a meeting.
2. System Architecture
The pipeline has four sequential stages. Each stage is a separate Python module so they can be run independently or end-to-end with a single command.
Stage
What it does
1. Ingest
Read leads from CSV/XLSX spreadsheet, validate and clean rows, output a structured leads list
2. Generate
For each lead, render the Jinja2 HTML template with their data, produce a unique HTML file per lead
3. Deploy
Screenshot each HTML with Playwright (for email embed), upload each HTML to Cloudflare R2 as a live hosted page
4. Send
For each lead, compose a personalized email with their demo URL and preview image, queue to Instantly.ai and fire

How to source the leads list

Option C — Manual: Build a targeted list of 500 specific businesses in one niche. Quality over quantity.
5. Stage 1 — Ingest Module
File: modules/ingest.py
Reads the spreadsheet, validates every row, normalizes data, and outputs leads_clean.json.
Function: run_ingest(input_path, output_path)
Accept .csv or .xlsx input — use pandas to read both
Lowercase all column headers, strip whitespace
Validate required fields (name, company, email, logo_url, niche) — skip rows missing any of these and log them to a file called skipped_leads.log
Validate email format with a basic regex: must contain @ and a dot after it
Validate logo_url — must start with http:// or https://
Generate a slug for each lead: lowercase company name, replace spaces with hyphens, strip special characters. Example: "Raza Electronics" → "raza-electronics"
Check for duplicate slugs — if collision, append a counter: raza-electronics-2
Output a JSON array to leads_clean.json where each object has: name, company, email, logo_url, niche, city, website, slug
Print summary: total rows, valid rows, skipped rows
6. Stage 2 — Generate Module
File: modules/generate.py
For each lead in leads_clean.json, renders the Jinja2 HTML template with their data and writes an index.html to output/demos/{slug}/.
Function: run_generate(leads_path, template_path, output_dir)
Load leads from leads_clean.json
Load the Jinja2 template from template/index.html
For each lead, render the template passing: name, company, logo_url, niche, city, slug, and a generated demo_url (see Section 8 for URL format)
Write rendered HTML to output/demos/{slug}/index.html
Also copy template/style.css and template/assets/ into each demo folder so it renders standalone
Print progress: "Generated demo 47/500 — raza-electronics"

7. The HTML Demo Template
File: template/index.html
This is the most important file — it is what the lead sees. It must look like a real, polished website. Use a clean professional design. The template should have exactly 3 sections:
Section 1 — Hero / Header
Display {{ logo_url }} as an <img> tag in the top-left, with a fallback placeholder if it fails to load
Large headline: "Here's what your {{ niche }} business could look like online"
Subheading: "We built this demo for {{ company }} in {{ city }} — it's ready to go live"
A primary CTA button: "Claim this website" linking to {{ claim_url }}
Section 2 — Services / Features
Three feature cards showing what the site includes
Use niche-aware copy — if niche is "restaurant" show: Online Menu, Reservation Booking, Delivery Integration. If niche is "salon" show: Services & Pricing, Online Booking, Gallery. Default copy for unknown niches: Professional Website, Mobile-Friendly Design, Contact & Enquiries
Implement this with a Jinja2 if/elif block for at least 5 niches: restaurant, salon, clothing, electronics, gym. One default fallback.
Section 3 — Footer / CTA
Repeat the CTA: "Ready to go live? Let's talk." with a button
Small text: "This demo was built specifically for {{ company }}. It is not live yet."
Your contact link or WhatsApp link
Design requirements for the template
Must be a single self-contained HTML file (inline CSS or linked style.css in same folder).
Must look good on mobile — use responsive CSS (flexbox or grid, no frameworks needed).
Logo image must use object-fit: contain and a fixed height so any logo shape looks fine.
Use a clean, modern font from Google Fonts — Poppins or Inter are fine here.
Color scheme: white background, dark text, one accent color (your brand color or a clean blue).
Do NOT use Bootstrap or heavy frameworks — keep it lightweight and fast.
8. Stage 3 — Deploy Module
File: modules/deploy.py
Two jobs: screenshot each demo as a PNG, then upload each demo folder to Cloudflare R2.
8a. Screenshot function: screenshot_demos(demos_dir, screenshots_dir)
Use playwright (sync API) with Chromium
For each lead slug, open output/demos/{slug}/index.html as a local file URL: file:///absolute/path/to/output/demos/{slug}/index.html
Set viewport to 1280x800
Wait for network idle before screenshot
Screenshot the full page, save to output/screenshots/{slug}.png
Resize/crop the PNG to 600px wide using Pillow — this is the email preview image
8b. Upload function: upload_to_r2(demos_dir, r2_bucket, r2_prefix)
Use boto3 with Cloudflare R2 credentials (R2 is S3-compatible)
R2 credentials come from .env: R2_ACCOUNT_ID, R2_ACCESS_KEY, R2_SECRET_KEY, R2_BUCKET_NAME, R2_PUBLIC_URL
For each lead slug, upload output/demos/{slug}/index.html to R2 key: {slug}/index.html
Also upload output/demos/{slug}/style.css to R2 key: {slug}/style.css
Upload any files in output/demos/{slug}/assets/ to R2 key: {slug}/assets/{filename}
Set ContentType correctly: text/html for .html, text/css for .css, image/png for .png, etc.
The live URL for each demo will be: {R2_PUBLIC_URL}/{slug} — store this on each lead object
Cloudflare R2 setup instructions for agent
1. Create a Cloudflare account and enable R2 storage.
2. Create a bucket named 'demos' (or whatever is in R2_BUCKET_NAME).
3. Create an R2 API token with read+write permissions.
4. Enable public access on the bucket and connect a custom domain (e.g. demos.yoursite.com).
5. In boto3, use endpoint_url = f'https://{R2_ACCOUNT_ID}.r2.cloudflarestorage.com'.
The boto3 S3 client works with R2 with no other changes.
9. Stage 4 — Send Module
File: modules/send.py
Composes and sends a personalized cold email to each lead via the Instantly.ai API.
Why Instantly.ai and not raw Resend/Mailgun
Sending 500 cold emails from a fresh domain via a transactional API will destroy deliverability and get the domain flagged as spam within days. Instantly.ai handles inbox warming, sending limits per day, reply detection, and follow-up sequences. Use their API to add leads and launch the campaign programmatically.
Function: run_send(leads_path, screenshots_dir)
Load leads from leads_clean.json
For each lead, compose the email using the template below
Upload the screenshot PNG to Instantly as the email's preview image (or host it on R2 and link as an <img> in HTML email body)
Add each lead to the Instantly campaign via their API: POST to /api/v1/lead/add
Include the personalized email body, subject, and the lead's demo URL
Rate limit: add a 0.5 second sleep between API calls to avoid rate limiting
Log success/failure for each lead to output/send_log.json
Email subject line
I built a website demo for {{ company }}
Email body (HTML)
Hi {{ name }},
I built a quick demo of what your {{ niche }} business could look like online.
Here it is: {{ demo_url }}
[SCREENSHOT IMAGE HERE — linked to demo_url]
It took me about 20 minutes to put together. I built it for {{ company }} specifically —
if you like it, I can have a full version live in 3 days.
Worth a quick chat?
[Your name]
[Your WhatsApp or calendar link]
Instantly.ai API setup
1. Create an Instantly.ai account and connect a warmed sending domain.
2. Create a campaign in the dashboard — note the campaign ID.
3. Get your API key from Settings > API.
4. Add to .env: INSTANTLY_API_KEY, INSTANTLY_CAMPAIGN_ID.
5. API endpoint to add a lead: POST https://api.instantly.ai/api/v1/lead/add
   Body: { api_key, campaign_id, email, first_name, company_name, personalization (the email body) }
10. Main Entry Point
File: main.py
Runs the full pipeline end-to-end with CLI flags to run individual stages.
Usage:
  python main.py --all              # Run all 4 stages
  python main.py --ingest           # Stage 1 only
  python main.py --generate         # Stage 2 only
  python main.py --deploy           # Stage 3 only (screenshots + R2 upload)
  python main.py --send             # Stage 4 only
  python main.py --screenshots-only # Stage 3a only, skip R2 upload
  python main.py --limit 10         # Process only first N leads (for testing)
The --limit flag is essential for testing. Always run --all --limit 5 first to verify the pipeline works end-to-end on a small batch before running all 500.

13. Error Handling Rules
Follow these rules throughout every module:
Every external call (R2 upload, Instantly API, logo URL fetch, Playwright screenshot) must be wrapped in try/except
On failure, log the error to a file (e.g. output/errors.log) with the lead slug and error message, then continue to the next lead — never crash the whole pipeline on a single lead failure
Logo URL failures: if the logo_url fails to load in the HTML template, fall back to a placeholder div with the company initials styled as a text logo
R2 upload failures: retry once after a 2-second wait, then log and skip
Instantly API failures: log to output/failed_sends.json so they can be retried separately
At the end of each stage, print a summary: success count, failure count, skipped count

15. What the Agent Should NOT Do
Do not use Flask, FastAPI, or any web server — this is a CLI script, not a web app
Do not use Docker — keep it simple, runs directly in Python
Do not use Selenium — use Playwright only
Do not use WidthType.PERCENTAGE anywhere — use DXA for all widths
Do not hardcode any API keys in source files — everything comes from .env
Do not send emails directly via SMTP or Resend for cold outreach — use Instantly.ai only
Do not overengineer — no databases, no queues, no async frameworks. Simple sequential Python is fine for 500 leads.
Do not skip the --limit flag for testing — always validate on small batches first
End of spec. Questions? Ask before building, not after.
